# C4 Architecture Documentation — Structurizr

## Overview

C4 is a model for describing software architecture at different levels of abstraction.
Dev-flow uses **Structurizr DSL** exclusively — no PlantUML. All architecture files
live in `docs/` at the project root.

## Directory Layout

```
docs/
├── workspace.dsl          # C4 model + !docs + !adrs references
├── workspace.json         # auto-generated by Structurizr Lite on first load — commit this
├── architecture/          # prose documentation sections (!docs)
│   ├── 01-overview.md
│   ├── 02-containers.md
│   ├── 03-components.md
│   ├── 04-key-flows.md
│   └── 05-development-guide.md
└── decisions/             # ADR files (!adrs)
    ├── 0001-approach-selection.md
    └── 0002-*.md
```

Mermaid sequence/flow diagrams remain in `.dev-flow/architecture/sequences/` and are
embedded by reference in `docs/architecture/04-key-flows.md`.

## workspace.dsl Structure

```structurizr
workspace {
    model {
        # Define people, software systems, containers, components
        user = person "User" "Description"
        system = softwareSystem "System Name" "Description" {
            webapp = container "Web App" "Description" "Technology"
            api    = container "API"     "Description" "Technology"
            db     = container "Database" "Description" "Technology"
        }

        user -> system "Uses"
        webapp -> api "Calls" "HTTPS/JSON"
        api -> db "Reads/writes" "SQL"
    }

    views {
        systemContext system "SystemContext" {
            include *
            autoLayout
        }

        container system "Containers" {
            include *
            autoLayout
        }

        component api "API-Components" {
            include *
            autoLayout
        }

        styles {
            element "Person" {
                shape Person
                background #08427b
                color #ffffff
            }
            element "Software System" {
                background #1168bd
                color #ffffff
            }
            element "Container" {
                background #438dd5
                color #ffffff
            }
            element "Component" {
                background #85bbf0
                color #000000
            }
            element "Database" {
                shape Cylinder
            }
        }
    }

    # Prose documentation — one .md file per section
    # These are workspace-level directives: place after the views {} block, inside workspace {}
    !docs architecture/

    # Architecture Decision Records — one .md file per decision
    !adrs decisions/
}
```

> **Note:** `!docs` and `!adrs` are workspace-level directives — they go inside `workspace { }` but outside both `model { }` and `views { }`. Place them after the closing brace of `views`.

## Starting Structurizr Lite

Run this one command:

```bash
docker run -it --rm -p 8080:8080 -v $(pwd)/docs:/usr/local/structurizr structurizr/lite
```

Open http://localhost:8080 in your browser.

Structurizr Lite watches `workspace.dsl` for changes and auto-refreshes the browser. Keep it running split-screen with your editor for the fastest feedback loop.

> **Note:** `workspace.json` is auto-generated by Structurizr Lite when it first reads `workspace.dsl`. After the first run, commit `workspace.json` to version control.

## Troubleshooting

### Common DSL Parse Errors

Structurizr DSL errors show the file, line number, unexpected token, and expected tokens. Here are the most common errors and how to fix them.

**Error: "Unexpected tokens (expected: include, exclude, autolayout, default, animation, title, description, properties)"**

Cause: A directive keyword used outside its valid context. The `replace` keyword, for example, is not valid at the workspace level.

Fix: Remove the invalid directive. Check the DSL reference for where a keyword is valid.

Example bad:
```
replace user [position:left]   # 'replace' is not valid at workspace level
```

Example good: Remove the line or move it inside the correct block.

---

**Error: "Unexpected tokens (expected: !docs, !decisions, group, container, description, tags, url, properties, perspectives, ->)"**

Cause: Incorrect syntax inside a `component` definition. The assignment operator (`=`) is not valid when defining inline components.

Example bad:
```
diComposition = component "DI Composition" "Single binding: LinkPort → adapter" "TypeScript" ""
```

Example good — no assignment operator for inline components:
```
component "DI Composition" "Single binding: LinkPort → adapter" "TypeScript"
```

Note: The fourth parameter (technology) is optional. An empty string `""` for technology is also unusual — omit it if there is no specific technology to note.

## The 4 C4 Levels

### Level 1: System Context

**In workspace.dsl model block:**
```
user = person "User" "Who uses the system"
system = softwareSystem "System Name" "What it does"
externalSystem = softwareSystem "External System" "What it provides" {
    tags "External"
}

user -> system "Uses"
system -> externalSystem "Calls"
```

**In views block:**
```
systemContext system "SystemContext" {
    include *
    autoLayout
}
```

**What to include:** Your system (one element), users (people), external systems with
relationship arrows and brief labels.

### Level 2: Container

**In workspace.dsl model block:**
```
system = softwareSystem "System" "Description" {
    webapp = container "Web App" "Serves the UI" "React / TypeScript"
    api    = container "API Server" "Handles requests" "Node.js / Express"
    db     = container "Database" "Persists data" "PostgreSQL"
    cache  = container "Cache" "Short-lived state" "Redis"

    webapp -> api   "API calls" "HTTPS/JSON"
    api    -> db    "Reads/writes" "SQL"
    api    -> cache "Reads/writes" "Redis protocol"
}
```

**In views block:**
```
container system "Containers" {
    include *
    autoLayout
}
```

### Level 3: Component

**In workspace.dsl model block** (inside a container):
```
api = container "API Server" "Handles requests" "Node.js" {
    authController    = component "Auth Controller"    "Handles auth flows"
    userService       = component "User Service"       "User business logic"
    userRepository    = component "User Repository"    "Data access"
    authMiddleware    = component "Auth Middleware"     "JWT validation"

    authController -> authMiddleware "Uses"
    authController -> userService   "Calls"
    userService    -> userRepository "Reads/writes"
}
```

**In views block:**
```
component api "API-Components" {
    include *
    autoLayout
}
```

The optional third parameter is the technology used: `component "Name" "Description" "Technology"`. Include it for components that have a notable technology (e.g., a specific library or framework). Omit it for pure logic components with no specific tech dependency.

**IMPORTANT**: Create a component view for EACH significant container. After the first,
ask: "Are there other containers that need component-level detail?" Loop until coverage
is complete.

### Level 4: Code (Optional)

Structurizr DSL does not have a native Level 4 element type. Level 4 is represented by the code itself. Skip Level 4 in this workflow unless there is a specific, documented reason to show class-level relationships — in that case, use a separate Mermaid class diagram in `.dev-flow/architecture/sequences/`.

## Documentation Sections (!docs)

Files in `docs/architecture/` are imported as documentation sections visible in
Structurizr's Documentation tab. Naming convention: `NN-kebab-title.md` where `NN` is
a two-digit sort prefix.

### Standard sections generated by Phase 3

| File | Content |
|------|---------|
| `01-overview.md` | What this system is, why it exists, key constraints, non-goals |
| `02-containers.md` | One prose paragraph per container — role, tech choice, key responsibilities |
| `03-components.md` | One paragraph per significant component — role and why it exists |
| `04-key-flows.md` | The core user journeys with embedded Mermaid diagram references |
| `05-development-guide.md` | How to run locally, env setup, common dev tasks |

These are **generated from Phase 3 artifacts** — not written from scratch. The walking
skeleton, C4 model, and folder structure that Phase 3 already produces are the source.

### Updating docs sections

During Phase 6: if implementation diverges from design, update the affected section.
During Phase 7: explicit "docs match reality" pass — update all sections.
During Phase 8: final review — every section must reflect the actual built system.

### Embedding diagrams

Reference Structurizr views inside docs using the embed syntax:

```markdown
![System Context](embed:SystemContext)
![Containers](embed:Containers)
```

## Architecture Decision Records (!adrs)

Files in `docs/decisions/` are imported as ADRs visible in Structurizr's Decisions tab.

### Naming convention

```
0001-approach-selection.md
0002-database-choice.md
0003-auth-strategy.md
```

Prefix `0001`, `0002`, etc. controls sort order. Structurizr renders status badges.

### ADR file format

```markdown
# N. Decision Title

Date: YYYY-MM-DD
Status: Accepted

## Context

What situation prompted this decision? What forces were at play?

## Decision

What was decided.

## Consequences

What becomes easier or harder as a result of this decision.
What was accepted as a trade-off.
```

**Status values:** Proposed | Accepted | Superseded | Deprecated | Rejected

### When to write an ADR

An ADR is written any time a significant architectural decision is made — one where
alternatives were considered and trade-offs were consciously accepted.

| Phase | Trigger | Example ADR (N = next number) |
|-------|---------|-------------------------------|
| 1 (Discovery) | Approach selected from brainstorm | `0001-approach-selection.md` |
| 3 (Design) | Technology/framework chosen | `0002-database-choice.md` |
| 3 (Design) | Deferred architectural decision documented | `0003-deferred-{name}.md` |
| 4 (Pre-Mortem) | A risk changes the design | `0004-risk-mitigation-{name}.md` |
| 6 (Implementation) | Design pivots mid-build | `0005-pivot-{name}.md` |
| 7 (Gap Analysis) | Gap fix changes the architecture | `0006-gap-fix-{name}.md` |

**Rule of thumb:** if a decision was significant enough to note down, it belongs in `docs/decisions/` as an ADR.
Each decision gets its own file — never append to an existing ADR.

If a later decision supersedes an earlier one, update the earlier ADR's status to
`Superseded by [ADR-0004](0004-new-decision.md)` and write the new ADR.

## C4 Iteration Process

C4 documentation is iterative, not one-shot:

1. Start with System Context and Container views
2. Identify the most critical/complex container
3. Create a Component view for that container
4. Ask: "Are there other containers that need component-level detail?"
5. If yes: add component block and view, repeat
6. If no: C4 model is complete for this phase

During Phase 6 implementation: update `workspace.dsl` if actual implementation differs
from the design. Keep the model and the code in sync.
