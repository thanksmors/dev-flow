# Phase 3: Design & Architecture

**Objective**: Create comprehensive design documents including C4 architecture, folder structure, walking skeleton definition, and elephant carpaccio vertical slices.

## Steps

### 3.1 Design the Walking Skeleton

Define the **thinnest possible end-to-end slice** that demonstrates:
- The system's basic structure works
- All major components connect
- Data flows from input to output
- Tests can run end-to-end

Document in `.dev-flow/architecture/walking-skeleton.md`:
- What the skeleton includes (and explicitly what it excludes)
- The end-to-end flow it demonstrates
- How it will be tested
- What "done" looks like for the skeleton

### 3.2 Define Vertical Slices (Elephant Carpaccio)

Break the feature into **thin vertical slices** using elephant carpaccio:

Read `${CLAUDE_PLUGIN_ROOT}/references/elephant-carpaccio.md` for the methodology.

For each slice, define:
- **Name**: What it delivers (user-visible value)
- **Scope**: What's included and excluded
- **Dependencies**: What other slices it depends on
- **Test strategy**: How it will be verified
- **Deferred decisions**: Architectural choices intentionally postponed

Order slices from smallest to largest, each building on the previous.

### 3.3 Design Folder Structure

**Canonical reference:** `references/layer-scaffold.md` — this defines the official Nuxt Layers directory structure, ports & adapters pattern, and naming conventions. Do not create a separate folder-structure doc.

For a new project (no existing structure):
1. Read `references/layer-scaffold.md` in full before proceeding
2. Create the `layers/` directory structure from the template
3. Create `layers/base/` for shared UI primitives
4. For each domain, create `layers/{domain}/` with the subdirectories defined in layer-scaffold
5. Create the DI composition root at `app/diComposition.ts`
6. Verify the structure compiles and the app starts

For an existing project (Phase 2 found conventions):
1. Read `references/layer-scaffold.md` in full
2. Compare the existing structure to the canonical structure
3. Document any deviations in `docs/decisions/` as ADRs (deviations must be justified)
4. If existing structure conflicts with layer-scaffold, propose a migration plan

**Ports & adapters:** Every external dependency (API, database, auth, observability) must have:
- A port interface in `layers/{domain}/ports/`
- A fake adapter in `layers/{domain}/adapters/` (permanent test fixture)
- A real adapter in `layers/{domain}/adapters/` (deferred decision)
- A single DI binding in `app/diComposition.ts`

See layer-scaffold.md for the complete ports+adapters wiring example.

### 3.4 C4 Architecture Documentation

Create C4 diagrams iteratively. Read `${CLAUDE_PLUGIN_ROOT}/references/c4-documentation.md` for full format guidance.

All Structurizr files go in `docs/` at the project root. Create the directory structure if it doesn't exist:

```
docs/
├── workspace.dsl
├── workspace.json         (auto-generated by Structurizr Lite on first load — commit it)
├── architecture/
└── decisions/
```

```
.dev-flow/architecture/
└── c4-coverage.md    (Component Coverage Checklist — enumerate-first output)
```

**Write `docs/workspace.dsl` with all four elements:**

1. **Model block** — define people, softwareSystems, containers, and components
2. **Views block** — systemContext, container, and component views with `autoLayout` and styles
3. **`!docs architecture/`** — connects the prose doc sections (written in step 3.8)
4. **`!adrs decisions/`** — connects the ADR files (first one written in step 3.7)

**Component loop (enumerate-first):**

Before creating any component views:
1. **List all containers** from the container view that has been defined
2. **For each container, enumerate its components** — read the codebase or feature spec and list every architecturally significant component (use `references/component-identification.md` scoring matrix if uncertain)
3. **Mark each component** as either:
   - Has a component view in this session, OR
   - Explicitly deferred (with one-line reason why)
4. **Then** create views for all non-deferred components

The output of this enumeration is a **Component Coverage Checklist** — save it to `.dev-flow/architecture/c4-coverage.md`. It becomes the record of what was documented and what was skipped.

Deferral is acceptable. Declaring "I considered all components and chose not to model X because Y" is better than accidentally omitting a container entirely.

**Container-First Enrichment Loop:**

Iterate through containers **one at a time**. For each container, fully enrich before moving to the next:

1. **Enrich the container** — read its current definition in workspace.dsl
   - Does it have a real name (not placeholder)?
   - Does it have a real description (not "TBD")?
   - Does it have a real technology (not "Node.js" if it's actually Python)?
   - Fix all placeholders before moving on

2. **Enrich each component inside this container** — for each component:
   - Real name (not `{ComponentName}`)
   - Real description (not "handles X" — describe what it actually does)
   - Real type (RequestProcessor, Entity, Repository, etc.)
   - Real technology if applicable
   - Relationships: are they bidirectional or unidirectional? Are the labels accurate?

3. **Add missing relationships** — after enriching, check if any component-to-component communication is missing from the DSL

4. **Verify in Structurizr Lite** — run `docker compose up`, open http://localhost:8080, verify the container view and its component view render correctly

5. **Commit this container's enrichment** — one commit per container, message like `feat(c4): enrich {ContainerName} container and components`

6. **Move to the next container** — repeat until all containers are enriched

**Rule:** Do not move to the next container until the current one is fully enriched. Partial enrichment is not acceptable — finish what you started.

**Note:** `workspace.json` is auto-generated by Structurizr Lite when it first reads `workspace.dsl`. Commit it after the first `docker compose up`.

### 3.5 State Diagrams

Create a **Mermaid stateDiagram** for any component or domain entity that has meaningful state with transitions. Read `${CLAUDE_PLUGIN_ROOT}/references/state-diagrams.md` for syntax and patterns.

**When to create a state diagram:**
- The domain has entities with distinct states (e.g., Order: `pending → processing → shipped → delivered`, Session: `anonymous → authenticated → expired`)
- A component has complex internal state machine behavior (e.g., AuthService: `idle → validating → authenticated → error`)
- A workflow has meaningful stages with transitions and guards (e.g., onboarding, checkout, approval)

**When to skip:**
- Entity is mostly a data container with CRUD transitions (just use the C4 component + repository)
- State is trivial (e.g., User: `active | suspended` — a boolean flag is clearer than a diagram)

**Trigger check:** After defining components in step 3.4, ask per component:
- "Does this have meaningful state with transitions?" → If yes, create a state diagram
- "Is the state complex enough that seeing it visually helps verify the implementation?" → If yes, create one

Place state diagrams in `.dev-flow/architecture/states/`. Use ` ```mermaid stateDiagram` format.

### 3.6 Sequence & Flow Diagrams

Create diagrams for key flows. Choose the right diagram type:

- **Mermaid sequenceDiagram** — for cross-container flows or simple 2-3 step interactions
- **Mermaid C4Dynamic** — for complex flows within a single container boundary that involve multiple steps with conditional branches, error handling, or compensating actions (see `references/c4-dynamic.md`)
- **Mermaid stateDiagram** — for state machine behavior (see `references/state-diagrams.md` and step 3.5)

For each flow, decide:
- Does the flow stay within one container? → C4Dynamic may be better
- Does it cross container boundaries? → sequenceDiagram is clearer
- Does it involve state transitions with guards/events? → stateDiagram
- Is it a simple request/response? → labeled Rel on a container view may suffice

- **Core flow diagram**: The main user journey / happy path
- **Error flow**: How errors propagate and are handled
- **C4Dynamic flow**: Complex flows within a container that involve multiple steps with conditional branches, error handling, or compensating actions (reference `references/c4-dynamic.md`)

Place in `.dev-flow/architecture/sequences/`. Use C4Dynamic (` ```mermaid C4Dynamic`) or sequenceDiagram format.

### 3.7 Deferred Architectural Decisions

Explicitly document decisions being **deferred**:

In `.dev-flow/architecture/deferred-decisions.md`:
```markdown
# Deferred Architectural Decisions

## Decision: {name}
- **Current choice**: {simple option, e.g., SQLite}
- **Future option**: {production option, e.g., PostgreSQL}
- **Adapter interface**: {interface that allows swapping}
- **When to swap**: {trigger for upgrading, e.g., "before production deploy"}
- **Complexity estimate**: {how hard the swap will be}
```

Each deferred decision MUST include an adapter/interface design so the swap is clean.

**IMPORTANT — populate the tracker immediately:** Every deferred decision identified in this phase (fake adapter, deferred DB choice, deferred auth provider, etc.) must be added to the tracker right now — not at Phase 6.5. The tracker is a living document from Phase 3 forward.

For each deferred decision in this phase:
1. Add a row to the tracker table
2. Set initial Status: `fake` or `pending`
3. Set the trigger criteria from what was documented above

### 3.8 Write Architecture Decision Records

For every significant design choice made in steps 3.1–3.7, write a separate ADR file in `docs/decisions/`.

Start with:

**`docs/decisions/0001-approach-selection.md`** — document the approach selected in Phase 1.

To find the approach: read `.dev-flow/design/approach.md` (written in Phase 1), or query Engram with `mem_search "{engramProjectKey}-phase1"` if the file is not present.

Then one ADR per major choice from this phase:
- Technology/framework selections
- Database choice (even if deferred — mark Status: Accepted with a note about deferral)
- Any other deferred architectural decisions (one ADR each)

See the ADR format in `${CLAUDE_PLUGIN_ROOT}/references/c4-documentation.md`.

Update `state.json` with all artifact paths.

### 3.8 Generate Documentation Sections

Create `docs/architecture/` with five sections derived from what Phase 3 already produced. Do NOT write these from scratch — extract from existing artifacts.

**`docs/architecture/01-overview.md`**
Source: Phase 1 brainstorm + selected approach + Phase 2 exploration findings.
Include:
- What this system is and why it exists
- Key constraints and non-goals
- The approach selected and why (brief — the ADR has the full reasoning)
- Diagram embed: `![System Context](embed:SystemContext)`

**`docs/architecture/02-containers.md`**
Source: `workspace.dsl` container block.
Include:
- One prose paragraph per container: role, technology choice, key responsibilities
- Diagram embed: `![Containers](embed:Containers)`

**`docs/architecture/03-components.md`**
Source: `workspace.dsl` component blocks.

For small systems (1–3 containers): generate one file as described above.

For larger systems (4+ containers): generate one file per container, named:
- `03a-components-{container-name}.md`

Each file:
- One section per significant component — what it does and why it exists as a separate component
- Diagram embed: `![{ContainerName} Components](embed:{ContainerName}-Components)`

The component coverage checklist (from Step 3.4) determines which containers get a separate file.

Note: As implementation proceeds, new components may be added. During Phase 6, update the component doc for the relevant container. During Phase 7, verify all components in the C4 model match the actual implementation.

**`docs/architecture/04-key-flows.md`**
Source: Mermaid diagrams in `.dev-flow/architecture/sequences/` and `.dev-flow/architecture/states/`.
Include:
- **Sequence diagrams** (from `.dev-flow/architecture/sequences/`): one section per flow (happy path, error path, data flow). Brief narrative before each diagram.
- **State diagrams** (from `.dev-flow/architecture/states/`): one section per state machine (entity lifecycle, service state, workflow stages). Brief narrative before each diagram.
- Link to each Mermaid diagram using a relative path reference rather than copying. For example:

  ```markdown
  See [Happy Path flow](../.dev-flow/architecture/sequences/happy-path.md) for the sequence diagram.
  See [Order entity lifecycle](../.dev-flow/architecture/states/order-entity.states.md) for the state diagram.
  ```

  Do NOT copy Mermaid source inline — keep a single source of truth in `.dev-flow/architecture/sequences/` and `.dev-flow/architecture/states/`.

**`docs/architecture/05-development-guide.md`**
Source: Walking skeleton definition + folder structure doc.
Include:
- Prerequisites (Docker, language runtime, etc.)
- How to start the dev environment. Write this based on what actually exists:
  - If the project has a `docker-compose.yml` with an app service AND the Structurizr service: "`docker compose up` (starts all services including Structurizr at http://localhost:8080)"
  - If `docker-compose.yml` only has the Structurizr service so far: document the app's own start command separately, then add "`docker compose up` (starts Structurizr at http://localhost:8080)"
  - If no docker-compose is used: document the standalone Docker command for Structurizr
- Folder structure overview (reference `.dev-flow/architecture/folder-structure.md`)
- How to run tests
- How to view architecture diagrams

**docker-compose setup:**

Add a `structurizr` service to the project's `docker-compose.yml` (create the file if it doesn't exist):

```yaml
  structurizr:
    image: structurizr/lite
    ports:
      - "8080:8080"
    volumes:
      - ./docs:/usr/local/structurizr
    restart: unless-stopped
```

If no docker-compose is appropriate for the project, document the standalone Docker fallback in `05-development-guide.md` instead:
```bash
docker run -it --rm -p 8080:8080 -v $(pwd)/docs:/usr/local/structurizr structurizr/lite
```

## User Manual Template

The user manual lives at `docs/superpowers/specs/YYYY-MM-DD-<project>-user-manual.md`.

Structure:
- **Overview**: what the app does, who uses it, core value (1 paragraph)
- **First-Time Setup**: step-by-step first-run experience
- **Core User Flows**: numbered steps for each major flow (what user does → what app shows)
- **Input Reference**: all user inputs with format and validation rules
- **Output Reference**: what the user sees after each action (success, error, empty state)
- **Error States**: what goes wrong and how to recover

The user manual is user-facing prose only. No code, no component names, no architecture.
It is a cross-reference artifact — acceptance criteria in the design spec reference
user manual sections, and Phase 6/7 verification traces back to user flows.

### Design Review with Frontend Commands

Before finalizing the container and component diagrams:

1. Run `/frontend-critique` on your component designs — does each component pass the AI Slop Test?
2. Check for common AI slop patterns:
   - Inter/Roboto defaults → use DM Sans + Fraunces
   - Centered card grids → try asymmetric layouts
   - Glassmorphism decoration → purposeful transparency only
   - Gray on colored → tint neutrals toward brand hue
   - Pure black/white → always tint
   - Bounce easing → use ease-out-quart

### README Outline

Before moving to the Pre-Mortem, sketch the project README:

1. **Project identity** — name, one-liner that answers "what problem does this solve"
2. **Section list** — ordered sections (e.g., What / Why / Quick Start / Architecture / Features / Contributing)
3. **Audience** — who is this README for; adjust tone and depth accordingly
4. **Brand voice** — casual, formal, playful; plain language or technical
5. **Differentiation** — what makes this project worth browsing vs. alternatives

Save the outline to `.dev-flow/design/readme-outline.md`. This file feeds the Phase 8 README draft.

> Skipping the README outline means the Phase 8 README is written without strategic intent — the "what story does this tell" question gets answered by whoever writes it last-minute.
